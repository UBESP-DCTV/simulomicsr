- **Status:** Approved (2026-05-02)
- **Phase:** P3.5-A (scaled benchmark, segue P3.5-B prototipo eval)
- **Approvals:** Luca Vedovelli (utente)
- **Riferimenti:** ADR-0006 §"Deliverable integrale: benchmark vs RummaGEO", spec P3.5-B `2026-05-02-p3.5-eval-benchmark-design.md`, spec classificatore v5 §6.2

## 1. Obiettivo

Validare il classificatore Stadio 2 di simulomicsr su un set indipendente di **100 GSE freschi** selezionati da `RummaGEO official ∩ xlsx gold`, stratificati per `design_kind` con oversampling delle categorie rare critiche per il claim del paper. Output: evidence base citabile direttamente in paper futuro.

P3.5-A è la prima fase del progetto progettata come **paper-ready evidence** (gate per P4 + materiale citabile in un futuro manoscritto). Riusa l'infrastruttura eval costruita in P3.5-B (`R/eval-stage2.R`, `R/eval-rummageo.R`, target eval pipeline) e aggiunge il layer di selezione GSE + investigation GSE145941 + statistica inferenziale (Wilson CI, McNemar test, bootstrap delta).

Il deliverable principale è **`analysis/eval/p35a-benchmark.html`** con 5 sezioni, di cui la Sezione 5 chiude la story dell'anomalia GSE145941 emersa in P3.5-B.

## 2. Razionale

### Perché paper-ready ora

L'utente ha confermato (2026-05-02) che **il paper è il deliverable principale del progetto**, insieme al pacchetto stesso. Rifare il benchmark più tardi (es. P5.5 dedicato) sarebbe spreco: il costo marginale di selezionare i 100 GSE con criterio paper-grade oggi è ~3-4 ore in più rispetto a un set "ragionevole" gate-only, contro il costo di rifare l'eval da zero più avanti.

### Perché 100 GSE freschi (no leakage)

I 15 GSE di P3 sono stati esposti al modello durante lo sviluppo del prompt Stage 2. Anche senza hard tuning, in un paper riportarli come parte del test set è data-leakage borderline. Il set di 100 GSE P3.5-A è **disgiunto** dai 15 P3, che restano citati come "scoping benchmark P3.5-B" — historical reference, non test set.

### Perché stratificazione hybrid (oversampling categorie rare)

Le categorie `factorial`, `time_course`, `mediated_effect` sono **dove simulomicsr dovrebbe battere RummaGEO** per costruzione (K-means+keyword di RummaGEO fallisce su quei pattern). Sotto-rappresentarle nel test set perderebbe potenza statistica esattamente sul claim più importante. La stratificazione hybrid usa keyword proxy per garantire copertura minima delle categorie rare; il resto è random sampling sul pool.

### Perché investigation GSE145941 livello B (non A o C)

P3.5-B ha avuto accuracy 41.7% sotto al casuale su `treatment_vs_untreated` (n=12), localizzata su GSE145941 (irradiazione 0Gy/10Gy, n=8, 12.5% accuracy). In un paper non si può scrivere "abbiamo un GSE outlier non investigato". Light (livello A) è informale; Deep+sampling (livello C) è prematuro — meglio capire il singolo caso, e se è un pattern lo ritroveremo automaticamente nei 100 nuovi via `eval_per_design_kind` e confusion matrix.

Livello B = re-classify GSE145941 con prompt verbose-reasoning + side-by-side con gold + commento entra come Sezione 5 del Quarto report.

## 5. Strategia di selezione GSE

### 5.1 Pool candidato

```
pool_candidates = RummaGEO_official_index  ∩  xlsx_gold_GSE  ∩  archs4_studies
```

- **RummaGEO official index**: scaricato da `rummageo.com` (API GraphQL o bulk dump). Stima ~22k studi RNA-seq.
- **xlsx_gold_GSE**: `unique(data-raw/relevant_sample_classified.xlsx$series_id)`. Stima ~6500 GSE.
- **archs4_studies**: filtro che il GSE sia presente in ARCHS4 H5 (necessario per Stadio 4-5 successivi). Sub-test: `archs4r::list_studies()` o equivalente.
- Stima pool: 1500-3000 GSE candidati. Sotto i 1500: rilassare il filtro ARCHS4 (deferred check) e procedere.

### 5.2 Keyword proxy per `design_kind`

Per ogni GSE candidato, classifica un `design_kind_proxy` da regex sui campi `string` xlsx (concatenati dei sample del GSE):

| Pattern keyword | design_kind_proxy |
|---|---|
| `\bsiRNA\b`, `\bshRNA\b`, `\bsi[A-Z][A-Z0-9]+\b` | `knockdown_panel` |
| `\b(\+\|-)\s*[Dd]ox\b`, `\binducible\b` + `[Dd]ox` | `knockdown_panel` (sub-tag `inducible` come metadata informativo) |
| `\b(\d+\s*[Gg]y)\b` con almeno due valori distinti (es. 0Gy + 10Gy) | `treatment_vs_untreated` |
| `\bDMSO\b`, `\bvehicle\b`, `\bcontrol\b` + drug/compound | `treatment_vs_vehicle` |
| `\b(\d+\s*[hd])\b` con ≥3 timepoint distinti | `time_course` |
| `\bknockout\b`, `\bKO\b`, `\b-/-\b`, `\bWT\b` | `knockout_vs_wt` |
| Almeno 2 fattori incrociati (es. `+drug` × `+siRNA`) | `factorial` |
| `\b(disease)\b`, `\b(cancer\|tumor\|patient)\b` + healthy/normal | `disease_vs_normal` |
| paracrine, conditioned media, transwell, co-culture | `mediated_effect` |
| nessun match | `unknown` (escluso da oversampling) |

Logica priorità: factorial > mediated_effect > time_course > inducible > knockdown/knockout > disease_vs_normal > treatment_vs_vehicle/untreated. Le categorie rare hanno priorità nel matching multi-keyword.

### 5.3 Stratified sampling — distribuzione target

| design_kind_proxy | Target N | Note |
|---|---|---|
| factorial | 15 | rare, critico per claim paper |
| time_course | 15 | rare, critico per claim paper |
| mediated_effect | 10 | molto rare, quanti il pool offre |
| treatment_vs_vehicle | 20 | base case |
| knockdown_panel | 15 | include inducible |
| disease_vs_normal | 15 | base case |
| treatment_vs_untreated | 10 | include irradiation patterns (verifica anomalia 145941) |
| **TOTALE** | **100** | |

Logica fallback: se una categoria ha < target candidati nel pool, prendi tutti i disponibili e ridistribuisci il deficit su `treatment_vs_vehicle` (categoria più popolata). Documentato in `inst/extdata/p35a-gse-selected.csv` con conteggi finali per categoria.

Seed deterministico: `set.seed(1812)` (stesso seed di P2 per consistenza).

### 5.4 Output committato

`inst/extdata/p35a-gse-selected.csv` contiene:
- `gse_id`
- `design_kind_proxy` (categoria dalla §5.2)
- `n_samples_xlsx` (count sample con gold in xlsx)
- `in_rummageo_official` (TRUE per costruzione)
- `included_in_archs4` (TRUE/FALSE)

Committato per riproducibilità del set: chi clona il repo può rieseguire la pipeline sullo stesso 100 GSE.

## 6. Investigation GSE145941 (Sezione 5)

### 6.1 Workflow

1. **Lettura artefatti P3.5-B**: `study_designs_validated[GSE145941]`, `sample_facts.stage1.v3.json` per gli 8 GSM, gold xlsx per gli 8 GSM.
2. **Re-classify con verbose reasoning**: `reclassify_verbose(GSE145941, prompt_extended)` — prompt esteso che chiede esplicitamente al modello di esplicitare:
   - assegnazione GSM → replicate_group con motivazione
   - identificazione del control/treatment arm con motivazione
   - inferenza `design_kind` con motivazione
3. **Verifica metadata GEO ufficiali**: fetch via `fetch_study_summary(GSE145941)` e confronto con string xlsx (potrebbe rivelare un mismatch dei metadata di provenienza diversa).
4. **Side-by-side report**: tabella con colonne `gsm_id | gold_xlsx | simulomicsr_p3 | simulomicsr_reclassify | archs4_metadata | conclusione`.
5. **Conclusione tracciabile**: una di {`gold_xlsx_error`, `llm_inference_error`, `metadata_mismatch_xlsx_vs_archs4`, `ambiguous_design`}, con razionale.

### 6.2 Cost

~1 chiamata LLM verbose (gpt-5.5, ~$0.30-0.50). Cached.

### 6.3 Outcome attesi

- Se `gold_xlsx_error`: documenta nel report, escludi GSE145941 dalla metrica retrospettiva, **non** è un bug di simulomicsr.
- Se `llm_inference_error`: documenta come failure mode noto del classificatore; cita nel paper come limitation; eventuale prompt fix è scope **P3.5-A2** (separato).
- Se `metadata_mismatch`: bug della pipeline upstream (xlsx vs ARCHS4); apri ADR per fix futuro.
- Se `ambiguous_design`: caso limite legittimo; cita nel paper come edge case.

## 7. Statistica inferenziale (esteso vs P3.5-B)

### 7.1 Confidence intervals

- **Accuracy globale**: Wilson 95% CI (preferito a normal approx per proporzioni vicine ai bordi).
- **Accuracy per_design_kind**: Wilson 95% CI per ogni design_kind.
- **Delta (simulomicsr − RummaGEO)**: bootstrap CI 95% (1000 resamples).

### 7.2 Test inferenziali

- **McNemar test paired** sui disagreement simulomicsr/RummaGEO sample-level (sample dove simulomicsr dice X e RummaGEO dice Y, vs gold).
- **Per_design_kind McNemar**: stesso test stratificato per `design_kind`. Reporting separato per categorie con n ≥ 25 (potenza adeguata); per categorie con n < 25, riporta valore con caveat "exploratory only, low power".
- **Multiple testing correction**: Holm su McNemar per_design_kind se ≥ 5 test paralleli (più potente di Bonferroni a parità di FWER).

### 7.3 Soglie di success

| Metrica | Soglia P3.5-A | Razionale |
|---|---|---|
| Accuracy globale | ≥ 80% | sopra P3.5-B (75.1%) ma realistico |
| Accuracy `treatment_vs_vehicle` | ≥ 85% | base case "facile" |
| Accuracy `treatment_vs_untreated` | ≥ 85% | base case "facile" (post-fix anomalia 145941) |
| Accuracy `factorial`/`time_course`/`mediated_effect` | ≥ 70% | task hard, sufficient se batte RummaGEO |
| McNemar p-value vs RummaGEO globale | < 0.05 | claim paper richiede significatività |
| McNemar p-value su `factorial` | < 0.05 | claim "LLM batte K-means" mirato |

Sotto soglia → investigation case-by-case; iter prompt è opzione (scope **P3.5-A2** separato, non P3.5-A).

## 8. Quarto report — 5 sezioni

### Sezione 1 — Stage 2 binary accuracy

- Tabella accuracy globale + Wilson CI (n totale, accuracy, sensitivity, specificity, F1, CI).
- Confusion matrix `simulomicsr_predicted × gold_binary`.
- Breakdown per `design_kind` con Wilson CI per categoria.
- Headline: "simulomicsr trtctr_predicted vs gold = X% (95% CI [a, b]) su n=~1500 sample, 100 GSE indipendenti".

### Sezione 2 — Granularity disagreement enumeration

- Riusata da P3.5-B, scalata. Conteggio sample per ogni design_role granulare (`negative_inducer_control`, `secondary_arm`, `bystander`, `baseline_t0`).
- Tabella con esempi rappresentativi per ogni categoria.

### Sezione 3 — Anchor coverage cross-studio (intra-100)

- Numero anchor unici nei 100 GSE.
- Anchor con N≥2 GSE → quante "potential meta-analyses" emergono dal subset.
- Grafico distribuzione N studi/anchor (atteso più dense di P3.5-B).
- Headline: "M anchor in N≥2 studi su 100 GSE; estrapolando a ~22k studi RummaGEO, atteso ~M*K dove K = scaling factor".

### Sezione 4 — RummaGEO head-to-head (con statistica)

- Confusion matrices: `RummaGEO × gold`, `simulomicsr × gold`, `simulomicsr × RummaGEO`.
- Accuracy globale RummaGEO (Wilson CI).
- Accuracy globale simulomicsr (Wilson CI).
- Delta accuracy con bootstrap CI 95%.
- McNemar test globale + p-value.
- McNemar test per_design_kind con multiple-testing correction.
- Headline: "simulomicsr beats RummaGEO by Δ pp (95% CI [a, b], McNemar p = ...) overall; on factorial designs Δ = X pp (p < 0.01)".

### Sezione 5 — Investigation GSE145941 (NUOVA)

- Tabella side-by-side gsm_id × {gold, simulomicsr_p3, simulomicsr_reclassify_verbose, archs4_metadata}.
- Estratto chain-of-thought verbose dal re-classify.
- Conclusione: una di {`gold_xlsx_error`, `llm_inference_error`, `metadata_mismatch`, `ambiguous_design`}.
- Implicazioni per il paper: limitation noto, fix scope, o validazione del classificatore.

## 9. Error handling

| Errore | Strategia |
|---|---|
| RummaGEO index download fail | abort early, escalate scelta A→B→C analoga a P3.5-B §6 |
| Pool candidati < 500 GSE | log warning; rilassa filtro ARCHS4; se ancora < 500 abort + decisione utente |
| Una categoria ha 0 candidati | log warning; ridistribuisci deficit su `treatment_vs_vehicle`; documenta nel CSV finale |
| Stage 1 LLM fail su singolo sample | escluso, conteggiato in tabella "Stage 1 validation failure rate" del report |
| Stage 2 LLM fail su singolo GSE | escluso, conteggiato in tabella "Stage 2 validation failure rate"; se >10% dei 100 GSE escalate |
| GSE escluso post-validazione | conteggio effettivo n GSE/sample sempre riportato nel report |
| Re-classify GSE145941 fail | report Sezione 5 vuoto + flag esplicito; non blocca le altre sezioni |
| Quarto build error | indipendente dai target; rebuild manuale dopo fix |
