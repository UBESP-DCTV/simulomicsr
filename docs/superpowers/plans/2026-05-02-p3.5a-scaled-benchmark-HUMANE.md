# P3.5-A — Scaled benchmark Stadio 2 (100 GSE paper-ready) — versione leggibile

Companion al plan dettagliato `2026-05-02-p3.5a-scaled-benchmark-plan.md`.
Qui solo cosa ti riguarda: cosa otterrai, dove devi decidere, dove la sessione si fermerà ad aspettarti.

## Cosa è P3.5-A in una frase

P3.5-B ha fatto la prova prototipo su 15 GSE già classificati. P3.5-A scala a **100 GSE freschi** selezionati a priori da `RummaGEO official ∩ xlsx gold` con stratificazione hybrid (oversampling delle categorie rare critiche per il paper) e produce **evidence base citabile direttamente nel paper futuro**: accuracy con Wilson CI, McNemar paired vs RummaGEO, bootstrap delta CI, breakdown per design_kind con Holm correction, investigation chiusa di GSE145941. Cost LLM ~$10-15.

## Cosa otterrai a fine P3.5-A

- `R/eval-stats.R` con **`wilson_ci()`**, **`mcnemar_paired()`**, **`bootstrap_delta_ci()`**, **`holm_adjust()`** — statistica inferenziale paper-grade. Tutte export pubbliche.
- `R/p35a-select-gse.R` con la pipeline di selezione GSE: indice RummaGEO via GraphQL paginated + cache, keyword proxy regex per `design_kind`, intersezione tre-vie (RummaGEO ∩ xlsx ∩ ARCHS4), stratified sampling deterministico (seed 1812).
- `R/p35a-investigate-gse145941.R` con **`reclassify_verbose()`** (re-classify con prompt extension che chiede chain-of-thought) e **`compare_with_gold()`** (tabella side-by-side per Sezione 5).
- **`inst/extdata/p35a-gse-selected.csv`** committato — i 100 GSE finali. Riproducibilità totale: chi clona il repo rilancia la pipeline sullo stesso set.
- Pipeline `analysis/_targets.R` estesa con ~15 nuovi target P3.5-A (suffisso `_p35a`), mantenendo intatto il flusso P3.5-B (i 15 GSE) per riferimento storico.
- **`analysis/eval/p35a-benchmark.html`** — il report finale con 5 sezioni:
  1. Stage 2 binary accuracy con Wilson 95% CI + breakdown per_design_kind con CI per categoria
  2. Granularity disagreement enumeration (atteso ~5-10× più dense di P3.5-B)
  3. Anchor coverage cross-studio intra-100 GSE
  4. RummaGEO head-to-head con McNemar test paired + bootstrap delta CI + Holm-corrected per_design_kind
  5. Investigation GSE145941 (chain-of-thought, side-by-side, diagnosi)

## Quello su cui devi decidere

### Adesso, prima di lanciare

**1. Distribuzione stratificazione target.** Il plan §5.3 della spec usa:

| design_kind_proxy | Target N |
|---|---|
| factorial | 15 |
| time_course | 15 |
| mediated_effect | 10 |
| treatment_vs_vehicle | 20 |
| knockdown_panel | 15 |
| disease_vs_normal | 15 |
| treatment_vs_untreated | 10 |

Se preferisci un mix diverso (più factorial / time_course per amplificare il claim del paper "LLM batte K-means su factorial"), cambia la tabella nel plan Step 5.6 prima di lanciare. Default raccomandato: tabella così com'è.

**2. Keyword proxy regex.** Il plan §4 ha la lista regex per design_kind_proxy. Se conosci pattern xlsx specifici che la mia regex non cattura, aggiungili in `R/p35a-select-gse.R::.DESIGN_KIND_PATTERNS` prima di lanciare. Default raccomandato: tabella così com'è — è proxy, non l'inferenza vera.

**3. ARCHS4 filter sul pool.** Il plan ha `archs4_studies = NULL` di default (no filter). Se vuoi forzare che i 100 GSE siano effettivamente in ARCHS4 (necessario per Stadio 4-5 successivi), aggiungi `archs4_studies = archs4r::list_studies()` allo Step 5.6. Default: skip — verifichiamo ARCHS4 al primo task di P5.

### Più avanti, durante l'esecuzione

**4. Soglia accuracy a fine Task 9 (run reale Stadio 1+2).**

- accuracy globale ≥ 80% → P3.5-A success. Sopra: procedi a Sezione 4 RummaGEO.
- accuracy globale < 80% → STOP CHECKPOINT. Decidiamo se iterare sul prompt (apri P3.5-A2 separato) o accettare il risultato e documentarlo come limitation.

**5. Decisione post-Sezione 4 (Task 10.4).**

- McNemar p-value globale < 0.05 e delta > 0 → claim paper sostenibile. Procedi a chiusura.
- McNemar non significativo o simulomicsr peggiora RummaGEO → STOP CHECKPOINT. Apri P3.5-A2 prompt iter.

### Decisioni rinviate non bloccanti per P3.5-A

- **Re-tuning prompt Stage 2** — eventuale P3.5-A2 separato, post-numbers.
- **Server-switch + run massivo ARCHS4** — P4, ADR-0005.
- **Migrazione a `ellmer`** — ADR separato pre-P4.
- **Rename pacchetto** — ADR-0003.
- **Effect-size pooling REM** — P5.

## Cosa P3.5-A NON fa (per evitare confusione)

- Non re-classifica i 15 GSE di P3.5-B — quelli restano historical reference.
- Non implementa effect-size meta-analysis (REM via metafor) — P5.
- Non costruisce gold "design-aware" separato — `trtctr_EP` xlsx binary è gold sufficiente.
- Non prova a "battere" RummaGEO modificando il prompt — solo misura. Il prompt iter è scope P3.5-A2 separato.
- Non tocca i target P3.5-B esistenti in `_targets.R` — solo append di nuovi target con suffisso `_p35a`.

## Stima di tempo

Esecuzione subagent-driven con review tra task, **12 task**. Stima realistica:

- **Walltime end-to-end**: ~mezza giornata (4-6 ore wallclock).
  - ~10-15 min: fetch RummaGEO index + selezione GSE
  - ~5-15 min: fetch GEO summaries via rentrez
  - ~30-90 min: Stadio 1 LLM (~2000 sample gpt-5.5)
  - ~15-45 min: Stadio 2 LLM (100 GSE gpt-5.5)
  - ~30 sec: investigation GSE145941 reclassify
  - ~5-30 min: RummaGEO fetch live per i 100 GSE
  - ~30 sec: Quarto build
  - resto = scripting, test, commit, review tra task
- **Lavoro umano attivo**: ~5-7 ore (dipende da quanti checkpoint di review).
- **Cost LLM totale**: ~$10-15 (Stadio 1 ~$5-10, Stadio 2 ~$2-4, reclassify ~$0.50).

## Dove la sessione si fermerà ad aspettarti

Sei stop espliciti pre-pianificati nel plan:

- **Step 5.6** (RUN selezione GSE live): conferma prima di partire — fetch RummaGEO index 5-10 min.
- **Step 6.4** (RUN Stadio 1 live): conferma prima di partire — walltime 30-90 min, cost $5-10.
- **Step 7.4** (RUN Stadio 2 live): conferma prima di partire — walltime 15-45 min, cost $2-4.
- **Step 8.6** (RUN reclassify_verbose): conferma prima di partire — walltime 30 sec, cost ~$0.50.
- **Step 9.3** (sanity check metriche): se accuracy < 0.80 STOP, decisione utente.
- **Step 10.4** (sanity check report): se Sezione 4 mostra McNemar non significativo o simulomicsr peggio di RummaGEO STOP, decisione utente.

Per il resto la sessione fila: tutti gli altri task sono offline (test mockati, R puro, schema validation, scrittura plan/commit).

## Cosa NON fa il plan (rispetto a un'esecuzione "ottimale")

- Non usa OpenAI batch API per Stadio 1 (sarebbe 50% costo + più walltime asincrono). Motivazione: l'infra batch non è ancora implementata; per 2000 chiamate sync è accettabile (1-1.5 ore). Batch entrerà in P4 con ADR dedicato.
- Non parallelizza Stadio 1 con `future`/`crew`. Motivazione: rate limit OpenAI e cache filesystem semplice. P4 può aggiungerlo se utile.
- Non genera gold design-aware separato. Confermato: `trtctr_EP` binary è sufficiente; granularity è flag, non metrica.
